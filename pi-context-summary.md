# Pi 会话上下文简表

Pi 在一次会话里，到底会给模型带上哪些“上下文”。

## 先分三类

不是所有东西都算同一种“注入”。

1. 真的会进系统提示词
2. 会作为消息进入当前回合
3. 不一定进提示词，但会影响模型能做什么

## 一、真的会进系统提示词的

默认情况下，Pi 会把下面这些内容拼进系统提示词：

- 基础系统提示词
  默认是一段内置的 coding assistant 提示词。

- `SYSTEM.md`
  如果存在项目级 `.pi/SYSTEM.md`，它会覆盖默认系统提示词。
  如果没有项目级，才会看全局 `~/.pi/agent/SYSTEM.md`。

- `APPEND_SYSTEM.md`
  这是追加，不是覆盖。
  项目级 `.pi/APPEND_SYSTEM.md` 优先；没有才看全局 `~/.pi/agent/APPEND_SYSTEM.md`。

- `AGENTS.md` / `CLAUDE.md`
  会按发现顺序拼进去。
  默认会包含全局 `~/.pi/agent/AGENTS.md`，再加上从当前工作目录一路往上找到的项目规则文件。

- skills 列表
  注意，这里默认注入的不是 skill 全文。
  注入的是 skill 的名字、描述、文件位置，告诉模型“有这些技能可读，需要时用 `read` 去打开”。
  只有当 `read` 工具可用时，这段 skills 列表才会放进系统提示词。

- 当前日期

- 当前工作目录

## 二、会作为消息进入当前回合的

这些不属于系统提示词，但也会进入这次对话上下文：

- 会话历史消息
  包括之前的用户消息、助手回复、工具结果、压缩后的摘要等。

- 用户当前输入

- 扩展在 `before_agent_start` 里额外塞进来的自定义消息
  这类内容会作为消息加到这一轮里，不是直接改系统提示词。

- 扩展通过 `nextTurn` 方式排队的消息
  会在下一次用户发起回合时一并带上。

- `/skill:name` 展开的 skill 全文
  这时就不是“只给一个技能目录”了，而是把 skill 文件正文包成一个消息发给模型。

- prompt template 展开后的内容
  例如 `/xxx` 这类模板，展开后会变成普通用户输入内容进入本轮。

## 三、不一定进提示词，但会影响能力的

这些东西更像“运行时能力”，不等于都被塞进了提示词正文：

- 内置工具
  比如 `read`、`bash`、`edit`、`write`。
  工具名和简短说明会出现在系统提示词里。

- extension 注册的工具
  它们会进入当前可用工具集合。
  是否在系统提示词里出现，要看这个工具有没有提供 prompt snippet / guideline 一类的提示信息。
  但不管写不写进提示词，工具本身都可能可调用。

- extension 注册的命令
  这是命令能力，不会默认作为系统提示词正文注入。
  更多是运行时命令入口。

- extension 的事件钩子
  比如：
  - `context`：可以改发给模型的消息列表
  - `before_agent_start`：可以改本轮系统提示词，或插入自定义消息
  - `before_provider_request`：可以改请求载荷

- 模型、thinking level、active tools
  这些会影响这一轮怎么跑，但不等于都以一段自然语言说明写进提示词。

## 四、最容易混淆的几点

- skills 默认不是全文注入
  默认只注入目录，真正内容要靠模型自己用 `read` 打开。

- prompt templates 默认不是系统提示词的一部分
  它们是在用户输入阶段被展开。

- extension 不等于“自动注入一大段上下文”
  extension 更像“可注册能力 + 可拦截流程”。
  它能改上下文，但不是天然都会往提示词里塞文本。

- `AGENTS.md` 和 `SYSTEM.md` 不是一回事
  `SYSTEM.md` 是替换/追加系统提示词的主入口。
  `AGENTS.md` 更像项目规则附录，会被拼到 `Project Context` 段里。

## 五、一句话版本

如果只说最核心的，Pi 一次会话常见会带上的东西是：

- 系统提示词
- `SYSTEM.md` / `APPEND_SYSTEM.md`
- `AGENTS.md` / `CLAUDE.md`
- skills 清单
- 当前日期
- 当前工作目录
- 历史消息
- 当前用户输入
- 扩展在运行时额外插进去的消息或提示词修改
