# Pi Gateway 群聊上下文优化落地记录

## 背景

之前 `pi-gateway` 在处理飞书群聊消息时，会把群聊路由信息拼进每一条用户消息里：

```txt
当前对话来自飞书群聊。
群 chat_id: xxx
发送者 open_id: xxx
发送者 user_id: xxx
群消息接收模式: mention
请回复当前群聊，不要把回复写成私聊。

用户消息：
...
```

这个做法的问题是：

1. `chat_id`、接收模式、回复目标属于网关路由，不是用户语义。
2. 每条消息重复注入固定说明，会污染上下文。
3. 群成员只显示 `open_id`，模型很难自然判断发言人。

## 参考 OC 后的结论

OpenClaw 的处理方式更清晰：路由、群聊说明、用户正文分开处理。

1. 回复目标由代码决定，模型不需要看到 `chat_id`。
2. 每条群消息只告诉模型“谁说了什么”。
3. 群聊说明作为会话上下文写入，不混进每条用户正文。
4. 飞书发送者昵称按用户资料接口查询，而不是遍历群成员列表。

因此本次实现不采用 `im.chatMembers.get(chat_id + open_id)` 作为主方案，改为按 `open_id` 查询用户资料。

## 已落地方案

### 1. 发送者昵称解析

新增：

```txt
src/feishu/user-context.ts
```

职责：

1. 根据 `open_id` 调用飞书用户资料接口解析展示名。
2. 成功结果缓存 10 分钟。
3. 失败或空结果短时间缓存 1 分钟，避免同一用户连续触发重复查询。
4. 查询失败不阻塞消息处理，降级显示 `open_id`。

### 2. 群聊消息正文瘦身

调整：

```txt
src/app/prompt-service.ts
```

群聊消息现在只注入发言人和正文：

```txt
发送者：Andy
用户消息：
...
```

昵称解析失败时：

```txt
发送者：ou_xxx
用户消息：
...
```

以下内容不再进入每条用户消息：

- `群 chat_id`
- `发送者 user_id`
- `群消息接收模式`
- `请回复当前群聊，不要把回复写成私聊`

### 3. 群聊说明写入会话上下文

调整：

```txt
src/pi/sessions.ts
```

群聊或话题会话会写入一次隐藏上下文：

```txt
这是一个飞书群聊。不同消息可能来自不同群成员；每条用户消息前会标明发言人。回复会自动发送回当前群聊，直接回应当前发言人即可。
```

这条说明会随会话上下文提供给模型，但不会重复拼到每条用户消息里。

为了兼容已有群聊会话，恢复或打开旧会话时也会检查是否已经写过；没写过才补一次。

### 4. 启动接线

调整：

```txt
src/index.ts
```

启动时创建飞书发送者昵称解析器，并注入到 prompt service。

## 验收标准

1. 群聊消息前缀为 `发送者：昵称`，昵称失败时降级为 `open_id`。
2. 每条群聊消息不再重复出现 `chat_id`、接收模式、回复目标说明。
3. 群聊说明只作为会话上下文写入一次。
4. 老群聊会话在恢复后也能补上群聊说明。
5. 同一用户短时间多次发言不会重复查飞书用户资料。
6. 飞书接口失败不会影响主流程。

## 测试

已覆盖：

1. 群聊 prompt 只包含发送者和用户消息。
2. 群聊会话创建时写入群聊说明。
3. 飞书发送者昵称解析、缓存和失败降级。
