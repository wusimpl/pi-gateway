# Bug：飞书群聊定时任务在私聊任务列表中可见

## 状态

- 记录时间：2026-04-28
- 项目：pi-gateway
- 严重程度：中高
- 类型：定时任务作用域 / 会话隔离问题

## 现象

在飞书私聊中查询「当前定时任务」时，返回结果包含了群聊创建的定时任务。

私聊环境本应只展示当前私聊用户可管理的任务，但列表中出现了群聊任务：

- 任务名：`每日天气播报-永州珠海`
- 任务 ID：`cron_0a4c6e77039a`
- 群聊目标：`oc_89842958dee35c5269834854b1750b22`
- schedule：每天 07:00，`Asia/Shanghai`
- prompt 中明确要求「在飞书群聊 ... 播报湖南永州和广东珠海当天的天气」

## 复现步骤

1. 在飞书群聊中创建一个定时任务。
2. 回到与机器人的私聊。
3. 发送「当前定时任务」。
4. 私聊返回的任务列表中包含群聊任务。

## 期望行为

私聊查询定时任务时，只应展示：

- 当前私聊用户在私聊作用域下创建的任务；或
- 明确设计为用户全局可见的个人任务。

不应展示群聊作用域任务。

## 实际行为

私聊任务列表返回了群聊任务，说明任务查询的 scope 隔离存在问题。

## 初步判断

当前定时任务的查询可能只按 `openId` / `scopeKey` 过滤，而没有区分：

- 私聊作用域
- 群聊作用域
- 群聊 ID / chat ID
- 任务创建来源上下文

从返回数据看，群聊任务仍带有当前用户的 `openId` / `scopeKey`，导致它被私聊 list 查询命中。

## 风险

- 私聊中暴露群聊任务，造成上下文串台。
- 用户可能在私聊中误删、误执行群聊任务。
- 后续如果群聊任务 prompt 含敏感上下文，可能产生隐私风险。
- 定时任务管理体验混乱，难以判断任务属于哪个会话。

## 建议修复方向

1. 定时任务持久化时增加明确的作用域字段，例如：
   - `scopeType`: `dm` / `group`
   - `chatId`: 群聊 ID，仅群聊任务有值
   - `userOpenId`: 创建者或私聊用户 ID
2. 查询 list 时按当前会话上下文过滤：
   - 私聊：只返回 `scopeType = dm` 且 `userOpenId = 当前用户` 的任务。
   - 群聊：只返回 `scopeType = group` 且 `chatId = 当前群聊` 的任务。
3. 删除、立即执行等管理操作也应校验当前上下文是否有权限操作该任务。
4. 对历史任务做兼容迁移：
   - 没有 `scopeType` 的老任务可按已有字段推断。
   - 如果 prompt 或 metadata 中含群聊目标，应补齐为 group scope。

## 验收标准

- 在私聊查询任务时，不再出现群聊任务。
- 在群聊查询任务时，只出现该群聊的任务，不出现私聊任务或其他群任务。
- 私聊无法删除 / 立即执行群聊任务。
- 群聊无法删除 / 立即执行其他群或私聊任务。
